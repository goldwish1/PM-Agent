## 1. 痛点与动机

### 1.1 两类目标（同时成立）

| 类型 | 说明 |
|------|------|
| 学习目标 | 通过亲手造一个 Agent，熟练《从零到一造 Agent》全流程：循环、工具、上下文、错误处理、交付 |
| 应用目标 | 面向 PM，在工作卡住时能被推荐实用工具，并（对少数工具）直接开出可改的草稿 |

### 1.2 候选痛点

| 痛点 | 场景 | 现有解法 | 频率（自评） |
|------|------|----------|--------------|
| 不知道用哪套 PM 方法 | 启动、风险、干系人、延期等 | 翻书 / 搜模板 / 问同事 | 中～高 |
| 知道工具名但仍难开工 | 推荐了章程/风险登记册后仍空白 | 自己找模板慢慢填 | 中 |
| `pm-toolbox` 偏「指路」 | 已有 39 工具 + AI 推荐，但推荐后闭环弱 | 打开详情页自行填写 | 已有产品缺口 |

### 1.3 痛点验证结论（当前）

- **已确认**：基于本人（第一用户）真实工作与练手需求，痛点成立。
- **未做**：外部 PM 访谈 / 社媒验证。MVP 阶段以「自己天天能用起来」为验收，推广前再补普遍性验证。

## 2. 核心需求

**一句话**：当 PM 描述眼前的项目卡点时，Agent 能推荐最贴的工具，并对「项目章程 / 风险登记册」直接生成可编辑的 Markdown 草稿。

成立性自检：

| 标准 | 是否满足 |
|------|----------|
| 能解释用户行为 | 是：问诊 → 推荐 → 开模板 → 改草稿 → 带走文件 |
| 满足后痛点减轻 | 是：少「该用啥」和「空白页怎么起笔」两段摩擦 |

---

## 3. 竞品与差异化（轻量）

| 对照物 | 做法 | 差异化机会 |
|--------|------|------------|
| 通用 ChatGPT | 口头建议，无结构化工具库与可控循环 | 工具库约束 + 确定性导出 + 可观察循环 |
| `pm-toolbox` 现状 | 站点 + 推荐 API，偏知识库与跳转 | 升级为 **CLI Agent 工作台**：多轮澄清 + 工具调用 + Markdown 落地 |
| 各类「PM Copilot」产品 | 偏会议/文档大而全 | MVP 故意做窄：练手完整、范围小、可验收 |

**差异化一句话**：不是又一个聊天机器人，而是「带刹车的 Agent 循环 + 受约束的 PM 工具箱 + 可带走的草稿」。

---

## 4. 方案结论

### 4.1 产品形态

**B. 工作台助手**

1. 听问题（可追问 1～2 句）  
2. 从工具库推荐 1～3 个工具并说明理由  
3. 若用户选定「项目章程」或「风险登记册」→ 多轮收集信息 → 生成草稿  
4. 导出本地 Markdown

### 4.2 实现路径（已确认）

**方案 A：从零手写最小 Agent 循环（TypeScript CLI）**

- 自研：loop、tool registry、迭代上限、工具错误回传为「指令」  
- 不引入 AgentClaw / LangChain 作为 MVP 底座（避免练手被框架淹没）  
- 工具内容：复制/同步 `pm-toolbox` 的 `tools.json` 到本仓，保持自包含

### 4.3 工程落点

| 项 | 结论 |
|----|------|
| 仓库 | 独立新仓，建议名 `pm-agent` |
| 入口 | 命令行终端聊天 |
| 与 toolbox | 数据同源、代码独立；日后网页可再接同一 Agent API（非本 MVP） |

## 5. MVP 范围

### 5.1 做

| 能力 | 说明 |
|------|------|
| Agent 循环 | 感知→推理→行动→观察；迭代上限；循环过程可打印/可观测 |
| 工具（建议 6 个左右） | 列/搜工具、推荐、取详情、起草章程、起草风险登记册、导出 Markdown |
| 推荐覆盖 | 全库工具均可推荐（约 39 个） |
| 填写覆盖 | **仅** 项目章程 + 风险登记册 |
| 产出 | 本地 `.md` 文件 |

### 5.2 不做

- 长期记忆 / 多租户  
- 飞书 / Web / Telegram 等渠道  
- 其余工具的模板填写（含 WBS、矩阵等）  
- 子 Agent、浏览器自动化、复杂并行调度  
- 生产级部署清单（容器、备份等可留作后续阶段对照书第 13 章）

### 5.3 验收标准（练手 + 可用）

1. 在终端能多轮对话，并看到工具调用过程。  
2. 任意 PM 卡点描述 → 得到有理由的 1～3 个工具推荐。  
3. 对「项目章程」「风险登记册」能生成并落盘 Markdown，打开可继续改。  
4. 循环不会明显空转（有迭代上限 / 重复失败可停止）。

## 6. 关键用户路径（MVP）

```
用户在终端启动 CLI
  → 描述卡点（如「下周要立项，还没正式授权」）
  → Agent 可选澄清 1～2 问
  → 调用 recommend → 给出工具（含项目章程）与理由
  → 用户说「帮我起草项目章程」
  → Agent 用 draft_charter 多轮补字段
  → export_markdown → 本地生成 项目章程-xxx.md
  → 用户带走修改
```

风险路径同理，终点为风险登记册 Markdown。
