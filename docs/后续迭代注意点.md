1. 若要接入 Telegram/Web 等多通道，考虑给工具 `execute` 注入 `context`（如 promptUser/sendFile），与通道解耦，避免继续靠闭包绑死 CLI 与 Session。
2. 工具的 description 要简单明了，for agent。
3. 起草侧「吃材料」：从 @附件或粘贴中抽取字段填章程/风险登记册（本版附件只服务推荐）。
4. `@` 路径 Tab 补全（prompt_toolkit）。

## 对话上下文管理 · 第四、五层（待迭代）

> 前三层（tool 压缩 + 15 轮滑动窗口 + `SessionState` 结构化快照）见实现计划与 `src/pm_agent/agent/context.py`。  
> 以下在超长会话、大附件、或窗口 + 快照仍不够时再做。

### 背景：何时需要第四、五层

| 场景 | 前三层是否够用 | 缺口 |
|------|--------------|------|
| 普通多轮推荐/陪跑（<15 轮） | 通常够 | — |
| 陪跑很久但未写进 `consulting_notes` 的闲聊背景 | 可能丢 | 需要 LLM 摘要补洞 |
| 用户 `@` / 粘贴接近 128KB 上限 | 不够 | 需要 token 级预算闸门 |
| 窗口内 tool 仍多、总 prompt 逼近模型上限 | 不够 | 需要按 token 动态再裁 |

### 第四层：LLM 会话摘要

**目标**：窗口裁掉旧回合后，把「未沉淀进 `consulting_notes` / draft」的对话背景压缩成一段文本，避免模型失忆。

**设计要点**：

- `SessionState` 新增可选字段 `session_summary: str | None`（进程内，关进程即丢）。
- 触发时机（建议 OR 关系，在 `prepare_messages_for_api` 末尾判断）：
  - 滑动窗口 `dropped_count > 0`，且快照块不足以覆盖关键背景；或
  - 估算 API 视图仍超软阈值（见第五层）且已无法再缩窗口/tool。
- 摘要调用：
  - 输入：被裁掉的 `user`/`assistant` 消息（**不含** tool JSON，tool 细节应已在第一层压过或进结构化状态）。
  - 输出：200～400 字中文，聚焦「卡点、约束、已做决定、待办」。
  - 可用同模型或更便宜模型；**单独一次** `llm.complete`，不走 tool。
- 注入方式：在主 `system` 后追加 `【会话摘要】…`；与第三层快照并存时，摘要偏「叙事」，快照偏「结构化状态」。
- **不要**每轮都摘要（成本与延迟）；仅超阈值时触发，并缓存到 `session_summary`，后续回合复用直至再次裁切。

**实现注意**：

- 摘要 API 调用与主 Agent Loop 解耦，避免死循环（摘要函数禁止再走 `prepare_messages_for_api`）。
- FakeLLM 测试路径：可 stub 返回固定摘要字符串。
- `output/debug/turn-*.json` 可选记录 `summary_updated: true` 便于观测。

**验收**：

- 裁掉 5+ 轮后，用户问「我一开始说的预算多少」模型仍能答（在未写入 `consulting_notes` 的剧本下）。
- 未触发裁切时不产生额外 API 调用。

### 第五层：Token 预算闸门

**目标**：用统一 token 预算替代「固定 15 轮」作为最终兜底，防止单轮大附件或密集 tool 撑爆上下文。

**设计要点**：

- `ContextPolicy` 扩展（环境变量建议）：
  - `PMBOX_CONTEXT_BUDGET`：总 prompt 预算，默认如 `24000`（为 64k 窗口留回复空间）。
  - `PMBOX_CONTEXT_RESERVED_COMPLETION`：预留给模型输出，默认 `4096`。
  - `PMBOX_CONTEXT_RESERVED_FIXED`：system + tools 粗算常量，默认 `6000`（或用 `estimate_messages_chars` 实测校准）。
- `prepare_messages_for_api` 末尾循环：

```text
while estimate_tokens(msgs) > budget:
  1. 压缩窗口内最早已完成回合的 tool（更激进）
  2. 缩小 window_turns（临时 -1，不低于下限如 3）
  3. 若仍超且未摘要 → 触发第四层摘要并替换被裁历史
  4. 硬截断兜底：截断当前回合 user 附件块（最后手段，需终端提示用户）
```

- Token 估算：
  - MVP：`chars // 2`（复用 `debug_log.estimate_messages_chars`）。
  - 后续：用 API `usage.prompt_tokens` 反馈校准系数，或引入 `tiktoken`（仅当值得加依赖时）。
- **附件联动**：`cli_attach` / `cli_paste` 的 128KB 字节预算与 context token 预算分离；第五层超限时优先缩历史，再提示缩短粘贴，而不是静默截断用户材料。

**实现注意**：

- 预算闸门默认**关闭**或软启用（仅 `PMBOX_CONTEXT_BUDGET>0` 时生效），避免改变前三层默认行为。
- `PMBOX_DEBUG` 摘要行可增加 `budget_used=…/…`。
- 纯函数测试：构造超长 messages，断言迭代裁剪后 `estimate_tokens` 低于预算。

**验收**：

- 注入 100KB 粘贴 + 10 轮历史，API 视图不超配置预算（或明确提示用户）。
- `PMBOX_CONTEXT_BUDGET=0` 时行为与仅前三层一致。

### 建议实现顺序

1. 先观测前三层上线后的 `usage.prompt_tokens` 分布（`PMBOX_DEBUG` / debug dump）。
2. 第五层软预算（只告警不硬裁）→ 硬裁循环。
3. 第四层摘要（依赖第五层判断「仍超限」）。

### 明确不做（本迭代）

- 跨会话持久化摘要或压缩历史（仍符合 MVP「关进程即丢」）。
- 向量记忆 / RAG 式长期记忆。
