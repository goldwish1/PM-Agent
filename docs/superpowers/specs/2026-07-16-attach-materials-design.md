# 设计：CLI `@` 附带材料辅助推荐

**日期**：2026-07-16  
**状态**：已实现（见 plans/2026-07-16-attach-materials.md）  
**范围**：下一迭代（MVP 之上）；不改动可起草工具白名单

---

## 1. 背景与问题

当前 pmbox 主路径是：用户描述卡点 →（可选澄清 ≤5 轮）→ 库内推荐 1～3 个工具 → 可选起草章程/风险登记册并导出。

真实使用中，用户有时已有卡点相关材料（纪要、一页纸、摘录），但产品只按「对话填信息」设计。材料若能进入上下文，可减少盲目澄清、提高推荐相关性。

同时明确：**不需要** Coding Agent 式的通用 `read_file` / `shell` / `grep`，也不做自动扫描文件夹。

---

## 2. 目标

**一句话**：用户在输入中用 `@路径` 附带 `.md` / `.txt`；CLI 在发送前读入并展示附件状态；Agent 依据材料优先做库内推荐，少澄清。

### 2.1 成功标准

1. 输入含合法 `@./notes.md` 时，终端在进 Agent 前打印 `[attach]` 成功信息，且模型本轮能看到文件正文（或截断后正文）。
2. 有成功附件时，行为倾向「少问快荐」（prompt 引导）；澄清硬上限仍为 5。
3. 无 `@` 时行为与现网一致。
4. 非法路径/类型/编码失败时，该附件失败原因可见；其它成功附件与用户正文仍可继续（除非「仅有失败 `@` 且无正文」——见错误处理）。

### 2.2 非目标（本版不做）

- 自动扫描目录 / 猜测相关文件
- 通用 `read_file`、`shell`、`grep` 工具
- `@` 路径 Tab 补全（可后续加）
- `.docx` / `.pdf` 等非纯文本
- 从材料自动抽取并填入章程/风险登记册字段（材料本版服务**推荐**，不主打起草）
- 跨轮隐式记住未再 `@` 的附件内容

---

## 3. 用户体验

### 3.1 典型输入

```text
下周要立项 @./kickoff.md 不知道从哪下手
```

### 3.2 终端反馈（进 Loop 前）

```text
[attach] ok  kickoff.md  (12KB)
```

失败示例：

```text
[attach] fail ./secret.bin  仅支持 .md / .txt
```

### 3.3 与粘贴的关系

用户直接粘贴一大段文字（无 `@`）仍完全可用，走现有 messages 路径；本功能只增强「点名本地文件」场景。

---

## 4. 方案选择

采用 **方案 1：CLI 解析 `@` + 注入本轮 user 消息**。

| 方案 | 要点 | 结论 |
|------|------|------|
| 1. CLI 注入 | 发送前读文件、拼进 user 文本；不新增 Agent 工具 | **采用** |
| 2. Session 附件 + 可选工具 | 状态更复杂，MVP 偏重 | 不做 |
| 3. 主要靠 Agent `read_file` | 不像 Claude Code；易漏读 | 不做 |

路径策略：相对 **进程 cwd** 的相对路径 + 用户写出的**绝对路径**（用户主动 `@` 即授权读该文件）。不做「禁出项目根」沙箱。

---

## 5. `@` 解析规则

1. 在用户输入中匹配 `@路径`；支持引号路径：`@"我的 纪要.md"`。
2. 同一条消息可多个 `@`，按出现顺序加载。
3. **附件判定**（避免把邮箱等当成路径）：`@` 之后须像路径——含 `/`、以 `./` 或 `../` 开头，或显式以 `.md` / `.txt` 结尾（大小写不敏感）。否则不视为附件。
4. 从用户自然语言中移除**所有已识别**的 `@路径` 片段（不论后续读文件成败）；若移除后仅空白，则自然语言视为空，语义为「仅附材料，请据材料推荐」。
5. 整行以 `/` 开头的输入仍走元指令（`/help`、`/quit` 等），**不**解析 `@`。

---

## 6. 文件与体积边界

| 规则 | 约定 |
|------|------|
| 类型 | 仅普通文件，扩展名 `.md` / `.txt`（大小写不敏感） |
| 路径 | 相对 cwd 或绝对路径；可为 `..`；拒绝目录 |
| 编码 | UTF-8；解码失败则该附件失败 |
| 单文件 | 原文软上限 **64KB**；超出截断并标记 `truncated` |
| 单轮合计 | 成功注入正文合计预算 **128KB**；按序载入直至预算用尽，其余标记失败或未载入并提示 |
| 副作用 | 只读打开；不执行文件内容 |

截断后的正文末尾须有明确标记（例如 `\n\n…[内容已截断]`）。

---

## 7. 注入到 messages 的形态

CLI 组装后交给 `handle_user_turn` 的文本：

- **识别到的 `@路径` 一律从自然语言中剥离**（无论该附件后来成功还是失败），避免正文里残留 `@./a.md`。
- **无成功附件**：只保留剥离后的自然语言；不拼附件块。
- **有成功附件**：

```text
<剥离后的用户自然语言>

---
[附件 1] path=<解析后路径> name=<文件名> truncated=<true|false>
<正文>
---
[附件 2] ...
```

附件**不**写入 Session 长期结构；本轮注入即可。后续轮次若仍需同一文件，用户须再次 `@`。

---

## 8. Prompt 与澄清策略

在 `prompts.py` 增加规则（有/无附件可用同一段静态条文，由模型根据 user 消息是否含附件块自行判断）：

- 若本轮用户消息含「附件」块：优先依据材料理解卡点并调用 `recommend_tools`；仅当仍缺阶段/卡点类型等关键信息时，再问**至多 1 个**澄清问题。
- 澄清硬上限仍为 **5**（`clarify_count` 逻辑不变）；有附件是软偏好「少问快荐」，不把硬上限改成 1。

---

## 9. 架构与模块

```text
read_user_line()
  →（若非 / 元指令）resolve_attachments(raw, cwd, limits)
  → 打印每条 [attach]
  → 若「无正文且无成功附件」→ 提示用法，不进 loop
  → 否则 handle_user_turn(assembled_text, ...)
```

| 模块 | 职责 |
|------|------|
| `src/pm_agent/cli_attach.py`（新） | 解析、读文件、截断、组装；以纯函数为主 |
| `src/pm_agent/cli.py` | 调用 attach、打印状态、传入组装文本；更新 HELP/欢迎语一行说明 |
| `src/pm_agent/agent/prompts.py` | 材料优先推荐条文 |
| Tool Registry | **不新增**读盘类工具 |

可选后续（非本版）：`cli_input.py` 对 `@` 做路径补全。

---

## 10. 错误处理

| 情况 | 行为 |
|------|------|
| 文件不存在 / 非文件 / 不可读 | 该附件 `fail` + 原因；其它继续 |
| 扩展名不支持 | 同上 |
| UTF-8 解码失败 | 同上，提示另存为 UTF-8 |
| 超单文件/合计预算 | 截断或跳过后续并提示 |
| 全部 `@` 失败且无剩余自然语言 | 打印用法示例，**不进入** Agent loop |
| 部分失败但有正文或有成功附件 | 进入 loop |

## 11. 测试计划

`tests/test_cli_attach.py`（名称可微调）覆盖：

1. 多 `@`、引号路径、邮箱式 `@user@x.com` 不误判为附件  
2. 相对路径 / 绝对路径；目录拒绝；扩展名拒绝  
3. 单文件截断与合计预算  
4. 无附件时组装结果恒等（或等价于去噪后原文）  
5. 有附件时块结构稳定（含 `truncated` 标记）

## 13. 决策摘要（来自澄清）

| 议题 | 决策 |
|------|------|
| 主路径 | 仍是卡点 → 推荐；材料为加分项 |
| 材料用途 | 优先服务**推荐更准 / 少澄清** |
| 主动性 | 完全被动（用户不给不读） |
| 交互 | Claude Code 式 `@` |
| 路径范围 | cwd 相对 + 绝对路径 |
| 格式 | 仅 `.md` / `.txt` |
| 有材料后行为 | 材料优先推荐 + `[attach]` 可见 |
| 实现 | CLI 注入，不新增通用读盘工具 |
