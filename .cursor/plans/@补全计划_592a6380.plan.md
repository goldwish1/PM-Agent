# `@` 文件补全实现计划

**目标**
在现有 CLI 输入层补上 `@` 文件候选补全：当用户在交互终端里输入 `@` 路径前缀时，显示当前工作目录下递归扫描到的 `.md` / `.txt` 文件候选，并支持 Tab 补全为可直接发送的 `@路径`。

## 范围
- 保留现有 slash 命令补全：[`src/pm_agent/cli_input.py`](src/pm_agent/cli_input.py)
- 复用现有 `@` 语法与解析规则：[`src/pm_agent/cli_attach.py`](src/pm_agent/cli_attach.py)
- 第一版仅补全**文件**，不补全目录，不做多类型扩展，不做 TUI 级文件选择器
- 候选范围为**当前工作目录递归扫描**到的 `.md` / `.txt`

## 设计要点
- 在 [`src/pm_agent/cli_input.py`](src/pm_agent/cli_input.py) 把当前 `SlashCompleter` 升级为组合 completer：
  - 输入以 `/` 开头时，继续走现有 slash 命令补全
  - 光标前存在正在输入的 `@前缀` 时，切换到 `@` 文件补全
- `@` 补全只负责**输入体验**，不改变发送后的解析逻辑；发送后仍由 [`src/pm_agent/cli_attach.py`](src/pm_agent/cli_attach.py) 负责识别、读取和注入
- 候选值建议使用**相对 cwd 的路径**，这样与现有 `load_attachment(..., cwd=Path.cwd())` 逻辑天然一致
- 对包含空格的候选，补全结果自动写成 `@"path with spaces.md"`，与现有 `_QUOTED` 规则兼容

## 主要改动
- [`src/pm_agent/cli_input.py`](src/pm_agent/cli_input.py)
  - 提取 `@` 触发片段识别函数，如 `extract_attach_fragment()`
  - 新增文件候选收集与过滤函数，如 `list_attach_candidates()`
  - 新增 `AttachCompleter` 或统一 `PmboxCompleter`，在 slash / `@` 两种模式间切换
- [`tests/`](tests/) 新增或扩展测试文件（优先单独建 `tests/test_cli_input.py`）
  - 测 `@` 片段提取
  - 测候选过滤与排序
  - 测空格路径转 quoted 结果
  - 测 slash 补全行为未回归
- 文档
  - 更新 [`README.md`](README.md) 或 CLI help 文案，补一句“输入 `@` 后可 Tab 补全 `.md/.txt` 文件”
  - 在 [`doc/agent_learn.md`](doc/agent_learn.md) 记录该交互增强

## 测试策略
- 纯函数单测优先，避免重度依赖 prompt_toolkit 交互对象
- 重点覆盖：
  - `@` 在行中间时也能识别当前片段
  - 仅返回 `.md/.txt` 文件，不返回目录/其他后缀
  - 相对路径输出稳定
  - 带空格文件名补全为引号形式
  - `/help` 等 slash 补全不受影响

## 风险与边界
- 递归扫描 cwd 可能在大目录下偏慢；第一版可先在每次补全时扫描，若体验差再考虑缓存
- 因为只补全文件、不补全目录，超深路径的发现性不如 Claude Code，但实现简单、风险低
- 仅是输入体验增强，不会改变现有 `@` 附件安全边界与解析边界
