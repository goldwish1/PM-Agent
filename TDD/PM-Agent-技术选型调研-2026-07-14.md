## 1. 调研目标与范围

### 1.1 项目背景（摘要）

PM Agent 是一个**命令行 Agent**：用户描述项目卡点 → 从约 39 个项目管理工具中推荐 → 对「项目章程 / 风险登记册」多轮起草 → 导出本地 Markdown。产品约束为：

- **练手优先**：从零手写 Agent 循环（感知→推理→行动→观察），对照《从零到一造 Agent》
- **独立 TypeScript 仓库**，自带 `tools.json`
- **不做**：Web 前端、数据库、登录、长期记忆、多渠道、子 Agent

### 1.2 本次调研要回答的问题

1. 用什么语言 / 运行时做 CLI Agent？  
2. Agent 循环自己写，还是用现成 Agent 框架？  
3. 怎么调大模型、怎么做 Tool Calling？  
4. CLI 交互用什么库？  
5. 工具参数校验、配置与产物落盘用什么？  
6. MVP 要不要数据库？  

### 1.3 在线调研来源（2026-07）

| 主题 | 参考 |
|------|------|
| TS Agent 栈对比 | Vercel AI SDK vs OpenAI Agents SDK、Mastra/LangGraph 综述（2026） |
| 何时不该上框架 | Speakeasy：简单 tool loop 可跳过重型框架 |
| Tool 循环官方模式 | Anthropic「Build a tool-using agent」：while + messages 历史 |
| DeepSeek Tool Calls | 官方文档：OpenAI 兼容；`openai` SDK + `baseURL`；模型需执行回传 |
| 与既有资产对齐 | `pm-toolbox` 已用 DeepSeek / OpenAI 兼容调用 |

## 2. 技术选型对比分析

### 2.1 语言与运行时

| 选项 | 作用 | 优点 | 缺点 | 结论 |
|------|------|------|------|------|
| **TypeScript + Node.js** | CLI、循环、工具、文件 I/O | 与决策一致；类型安全；和 toolbox / 生态一致 | 需编译或 tsx | **采用** |
| Python | 同上 | Agent 教程多 | 偏离已确认「TS」决策；两套栈 | 不采用 |

**通俗解释**：选 TypeScript，是因为你已经定了「手写 TS 最小循环」，且以后若挂回 Web，同一语言成本最低。

---

### 2.2 Agent 编排层（核心决策）

| 选项 | 作用 | 为什么考虑 | 优点 | 缺点 | 结论 |
|------|------|------------|------|------|------|
| **自研 Agent Loop** | while：调模型 → 解析 tool_calls → 执行工具 → 把结果塞回 messages → 直到结束或触顶 | 对齐练手目标与书籍；PRD 要求循环可见、迭代上限 | 机制透明；易对照书里五层防护；不被框架淹没 | 要自己写错误分类、日志、上限 | **采用（推荐）** |
| Vercel AI SDK（`generateText` + maxSteps） | 封装多步 tool loop | Speakeasy 称简单 tool loop 可用它 | 省循环样板；多模型 | 「造循环」体感弱；练手目标打折 | 备选（后期可外包 I/O，不推荐 MVP） |
| OpenAI Agents SDK / Mastra / LangGraph.js | 完整 Agent 运行时/图编排 | 生产向、handoff/追踪 | 能力强 | 过重；子 Agent/图非本 MVP；练手被框架带走 | **不采用（MVP）** |
| Claude Agent SDK | 托管 Claude Code 同款循环 | 强工具生态 | 循环与权限现成 | 偏 Anthropic 锁定；距离「从零写循环」更远 | 不采用 |

**关键理由（用人话）**：  
市面文章普遍说——**工具只有几个、流程近似线性时，不必上 LangGraph/Mastra**。你要练的就是循环本身，所以 MVP 应自己写 50～150 行级的 loop，而不是 import 一个黑盒 Runner。

---

### 2.3 大模型调用与 Tool Calling

| 选项 | 作用 | 优点 | 缺点 | 结论 |
|------|------|------|------|------|
| **`openai` npm + DeepSeek `baseURL`** | Chat Completions + tools | 与 `pm-toolbox` 一致；官方支持 Tool Calls；换 OpenAI/兼容端几乎只改正文 | DeepSeek 模型 ID 需跟官方更新（现调研指向 v4 系列；兼容旧 alias） | **默认采用** |
| `@anthropic-ai/sdk` 直连 | Messages API + tool_use | 文档循环示例清晰 | 第二套 API 形状；练手期增加分支 | 可选第二 Provider（非必须） |
| 仅用 HTTP `fetch` 裸调 | 不依赖 SDK | 依赖最少 | 类型差、重试/流式要自己拼 | 不划算 |

**通俗解释**：

- DeepSeek **不会替你执行工具**，它只告诉你「该调哪个函数、参数是什么」；**你的 Agent 要真的跑工具，再把结果喂回去**——这正是 Agent 和聊天机器人的差别。  
- 用官方兼容的 `openai` 包，等于一把钥匙开多把锁（DeepSeek / OpenAI 兼容网关）。

**建议默认模型策略**（实现细节可在技术方案微调）：

- 主：DeepSeek（`DEEPSEEK_API_KEY`）  
- 备：`OPENAI_API_KEY`（同协议切换）  
- 配置用 `.env`，密钥永不写入 Markdown 导出

---

### 2.4 CLI 交互层

| 选项 | 作用 | 优点 | 缺点 | 结论 |
|------|------|------|------|------|
| **Node `readline` / `node:readline/promises`** | 读用户一行输入、打出回复 | 零花活；贴「最小 CLI」；学习成本低 | 无漂亮输入框等高级 UI | **采用** |
| Ink（React for CLI） | 花哨终端 UI | 好看 | 过重，偏离练手 | 不采用 |
| Commander / yargs | 子命令解析 | 适合 `pm-agent recommend` 子命令模式 | 本产品是**长对话**，不是命令工具箱 | 可仅做 `pm-agent` 启动入口，不做多子命令 |

---

### 2.5 工具定义与参数校验

| 选项 | 作用 | 优点 | 缺点 | 结论 |
|------|------|------|------|------|
| **手写 Tool Registry（name/description/parameters/execute）** | 对齐书中六字段思路 | 练工具设计；description 质量可控 | 要自己维护 schema | **采用** |
| **Zod** | 校验模型传来的 arguments | 防幻觉字段；与 DeepSeek 实践文章一致 | 多一个依赖 | **采用（轻量）** |
| MCP 服务化工具 | 进程外挂工具 | 扩展性好 | MVP 过重 | 不做 |

**建议工具集合（与 PRD 对齐，约 6 个）**：

| 工具名（示意） | 副作用 | 说明 |
|----------------|--------|------|
| `list_tools` / `search_tools` | 无（pure） | 浏览/检索工具库 |
| `recommend_tools` | 无 | 基于问题推荐 1～3（也可由模型+检索组合） |
| `get_tool_detail` | 无 | 详情 |
| `draft_project_charter` | 有（改会话草稿状态） | 更新章程草稿 |
| `draft_risk_register` | 有 | 更新风险草稿（1～3 条） |
| `export_markdown` | 有（写磁盘） | 仅允许写 `output/` |

---

### 2.6 数据与持久化

| 选项 | 作用 | 结论 |
|------|------|------|
| **内存 messages + 会话草稿对象** | 当次对话上下文与起草状态 | **采用**（关进程即丢，符合 PRD） |
| SQLite / 向量库记忆 | 跨会话记忆 | **不做**（PRD 明确排除） |
| 仅 `tools.json` + `output/*.md` 落盘 | 知识只读 + 产物写出 | **采用** |

---

### 2.7 工程工具链

| 选项 | 作用 | 结论 |
|------|------|------|
| TypeScript strict + `tsx` 直接跑 | 少配构建 | **采用** |
| `dotenv` | 读 API Key | **采用** |
| ESLint + Prettier | 风格统一 | 建议有，可第二阶段再硬性加 |
| Vitest | 测 tool execute、路径白名单、循环上限 | 建议有；先测纯函数层 |

## 3. 推荐技术栈总表（拟确认）

| 层级 | 选型 | 一句话理由 |
|------|------|------------|
| 语言/运行时 | TypeScript + Node.js 20+ | 已确认；类型安全 |
| Agent 编排 | **自研 Loop** + 迭代上限 + 可见日志 | 练手核心；对齐书与 PRD |
| LLM | `openai` SDK → DeepSeek（兼容 OpenAI） | 工具调用成熟；与 toolbox 一致 |
| 校验 | Zod | 拦住错误 tool 参数 |
| CLI | readline | 最小可用 |
| 数据 | `data/tools.json` + 内存会话 + `output/` | 无数据库也能闭环 |
| 配置 | `.env` + `.gitignore` | 密钥安全（PRD 6.3） |

---

## 4. 关键技术选型理由（汇总）

1. **自研循环，而不是 Agents/Mastra/LangGraph**  
   PRD 与用户目标都是「练造 Agent」；业界对「几个工具的线性 loop」也建议别上重型框架。

2. **OpenAI 兼容客户端 + DeepSeek**  
   Tool Calls 有官方支持与明确「应用侧执行工具」语义；与现有 `pm-toolbox` 习惯一致，降低联调成本。

3. **无 DB、无 Web、无 MCP**  
   把复杂度留给「循环 + 工具描述 + 导出」三条线，避免技术方案假装前后端分离却没有前端。

4. **Zod + output 路径白名单**  
   对应 PRD：幻觉参数、任意路径写入，都必须用确定性规则挡在工具层，而不是指望模型「自觉」。

## 5. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 模型乱造工具名 | 推荐结果用 slug 与 `tools.json` 交叉校验，非法则工具返回「纠正指令」 |
| DeepSeek 模型 ID 变更 | 配置项可换 model；文档写清环境变量 |
| 手写循环写歪 | 对照 Anthropic/DeepSeek 官方 tool 回传模式；设 `maxIterations` |
| 上下文塞满 tools.json | 分层：列表先摘要；详情再 `get_tool_detail`（对齐书「工具分层加载」） |
