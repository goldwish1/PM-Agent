## 1. 设计目标

在现有架构（自研 Agent Loop + Tool Registry + SessionState）基础上，新增一个**通用的陪跑讨论阶段**，衔接「推荐工具」与「起草文档」之间的断层，且不为不同文档类型（项目章程 / 风险登记册）重复设计逻辑。

设计必须满足需求文档中已确认的 6 条原则：个性化建议、Agent 主动提炼 + 预览确认闸门、显式状态机、不设讨论轮次上限、保留兜底路径、机制普适于所有 `draftable` 工具。

## 2. 会话状态机改造

### 2.1 现状（`src/pm_agent/agent/session.py`）

```python
class SessionMode(StrEnum):
    IDLE = "idle"
    CLARIFYING = "clarifying"
    RECOMMENDING = "recommending"
    DRAFTING_CHARTER = "drafting_charter"
    DRAFTING_RISK = "drafting_risk"
    PREVIEW = "preview"
```

### 2.2 调整后

```python
class SessionMode(StrEnum):
    IDLE = "idle"
    CLARIFYING = "clarifying"
    RECOMMENDING = "recommending"
    CONSULTING = "consulting"        # 新增：推荐后的陪跑讨论态，通用于所有 draftable 工具
    DRAFTING_CHARTER = "drafting_charter"
    DRAFTING_RISK = "drafting_risk"
    PREVIEW = "preview"
```

状态流转：

```
IDLE → CLARIFYING → RECOMMENDING → CONSULTING → DRAFTING_* → PREVIEW
                          │                           ▲
                          └───────（兜底：跳过陪跑）────┘
```

- `RECOMMENDING` → 用户对某个 `draftable: true` 工具表达"怎么用"/希望讨论 → 进入 `CONSULTING`。
- `RECOMMENDING`（或 `CONSULTING`）→ 用户明确要求起草 → 进入对应 `DRAFTING_*`（兜底路径直接跳过 `CONSULTING`）。
- `DRAFTING_CHARTER` / `DRAFTING_RISK` 两个模式保留不变，因为落盘文档结构天然不同（8 字段 vs 风险条目列表），但其**字段值来源**统一改为「`consulting_notes` + 全量 `messages` 提炼」，而非逐项追问用户（兜底路径除外）。

### 2.3 `SessionState` 新增字段

```python
@dataclass
class SessionState:
    ...
    consulting_tool_slug: str | None = None          # 当前陪跑讨论聚焦的工具 slug
    consulting_notes: list[str] = field(default_factory=list)  # 讨论中沉淀的结构化事实
```

- `consulting_tool_slug`：记录陪跑讨论围绕哪个 `draftable` 工具展开，供起草阶段判断该调用哪个 `draft_*` 工具。
- `consulting_notes`：讨论过程中由 LLM 主动调用工具追加的关键事实列表，作为起草阶段字段提炼的结构化输入来源，避免完全依赖模型对长对话历史的自行回忆。

## 3. 新增工具（通用，不区分文档类型）

新增子包 `src/pm_agent/tools/consult/`，与现有 `knowledge/`、`draft/`、`export/`、`demo/` 平级。

### 3.1 `start_consulting`

- **用途**：从 `RECOMMENDING` 进入 `CONSULTING`；记录当前讨论聚焦的工具。
- **参数**：`tool_slug: str`（必须是库内 `draftable: true` 的工具，非法/不可起草工具需拒绝并说明原因）。
- **行为**：
  1. 校验 `tool_slug` 存在且 `draftable == true`；否则返回错误说明（沿用现状「MVP 仅支持章程/风险登记册起草」的拒绝话术）。
  2. `state.mode = SessionMode.CONSULTING`；`state.consulting_tool_slug = tool_slug`。
  3. 返回该工具的 `steps` / `scenarios` 作为**素材**，供 LLM 在后续追问中结合用户情境展开，而不是直接把这些字段转述给用户。
- **不做的事**：不负责生成个性化建议内容本身——个性化建议是 LLM 在系统提示词约束下自行组织语言的职责，本工具只提供确定性的素材与状态切换。

### 3.2 `note_consulting_fact`

- **用途**：讨论过程中，LLM 判断某句话是有价值的项目背景信息时，调用本工具结构化沉淀。
- **参数**：`fact: str`（简洁的关键事实描述，例如"预算约 50 万，需在 8 月底前完成立项"）。
- **行为**：
  1. 仅在 `state.mode == CONSULTING` 时允许调用；否则返回提示（避免误用）。
  2. `state.consulting_notes.append(fact)`。
  3. 返回当前已沉淀的 `consulting_notes` 数量，便于 LLM 自我感知讨论沉淀进度。
- **设计动机**：把"沉淀"这个动作显式工程化为一次工具调用，而不是仅仅依赖 session mode 切换或 prompt 提醒——防止长对话中模型注意力衰减导致讨论内容丢失，这是解决需求文档中断层 3 的关键手段。

两个工具均标记 `category="consult"`，`pure=False`（有状态副作用，不适合并行执行）。

## 4. 起草工具的调用策略调整（参数结构不变）

`draft_project_charter`（`src/pm_agent/tools/draft/charter.py`）与 `draft_risk_register`（`src/pm_agent/tools/draft/risk.py`）的 **Pydantic 参数模型不做结构性改动**（仍是 8 个可选字段 / 风险条目列表 + 增量合并语义），改变的是 **LLM 的调用策略**，通过系统提示词约束实现：

- 若 `state.consulting_notes` 非空（即经过陪跑讨论）：LLM 应先基于 `consulting_notes` + 对话历史，一次性提炼出尽可能完整的字段候选值，再调用 `draft_project_charter` / `draft_risk_register` 提交，而不是重新逐字段/逐条询问用户。
- 现有 `missing_fields()` / `preview_lines()` 预览机制不变：用户看到预览后若有字段不准确，可直接说"XX 字段应该是……"，走现有的增量合并逻辑修正，而不是从头填空。
- 若未经陪跑讨论（`consulting_notes` 为空）用户直接要求起草：沿用现状的逐字段/逐条追问兜底路径，不受影响。

这样两个起草工具的代码本身几乎不需要改动（除非需要在 payload 中额外提示"当前是否有 consulting_notes 可用"），核心改动集中在 `prompts.py` 的指令层面。

## 5. 系统提示词调整（`src/pm_agent/agent/prompts.py`）

新增一段**通用规则**（不为章程/风险登记册分别写话术），插入到 `SYSTEM_PROMPT` 的"工具使用优先级"与"硬性约束"之间，大致方向：

```
## 陪跑讨论（CONSULTING）
- 当用户对某个可起草工具（draftable=true）表达"怎么用"或希望深入讨论使用思路时，
  调用 start_consulting 进入陪跑模式。
- 陪跑模式下：结合该工具的 steps/scenarios，主动追问用户的具体情境
  （项目背景、约束、干系人、预算、时间压力等），给出针对该情境的个性化建议，
  禁止仅复述通用步骤说明。
- 讨论中一旦获得有价值的项目背景信息，调用 note_consulting_fact 沉淀，
  不要仅依赖对话历史等待起草时再回忆。
- 不设讨论轮次上限；当判断讨论已经比较充分时，可主动询问用户
  "是否可以开始起草"，但不得强制打断用户继续讨论。
- 用户明确要求起草时：若已有 consulting_notes，基于其内容 + 对话历史
  一次性提炼候选字段值并提交 draft_*工具，展示预览请用户确认或修正；
  若没有陪跑讨论记录，沿用逐字段/逐条追问的方式起草，不强制要求先讨论。
```

该规则写在通用位置，同时适用于「项目章程」与「风险登记册」两个 `draftable` 工具，未来新增第三个可起草工具时无需扩写。

## 7. 测试要点

1. `start_consulting`：
   - 对 `draftable: true` 工具（`project-charter`、`risk-register`）应成功切换状态并记录 `consulting_tool_slug`。
   - 对 `draftable: false` 工具应拒绝并返回说明，不改变 `state.mode`。
   - 对非法/库外 slug 应拒绝，不得编造工具信息。

2. `note_consulting_fact`：
   - 非 `CONSULTING` 状态下调用应被拒绝或提示，不追加内容。
   - `CONSULTING` 状态下多次调用应正确累加 `consulting_notes`，不覆盖历史条目。

3. 起草流程回归：
   - 现有「无陪跑讨论直接起草」路径（`tests/test_draft_merge.py` 等）应保持通过，验证兜底路径未被破坏。
   - 新增用例验证：`consulting_notes` 非空时，`draft_project_charter` 收到的字段值应能对应到之前沉淀的事实（通过模拟 LLM 提炼后传参的方式验证参数正确合并，而非验证 LLM 的提炼能力本身——后者依赖模型层，不在单测范围）。

4. 状态机边界：
   - `CONSULTING` 与 `DRAFTING_*` 之间的切换在多种触发路径下（陪跑后起草 / 兜底直接起草）都不应导致状态错乱或 `consulting_notes` 丢失。

## 8. 后续可选增强（不在本次范围内）

- 若后续要支持更多 `draftable` 工具，本设计已经是通用机制，预期无需改动核心状态机或工具接口，只需在 `data/tools.json` 中标记 `draftable: true` 并补充对应 `draft_*` 工具。
- 若发现 `consulting_notes` 简单字符串列表在长讨论中仍不足以支撑高质量提炼，可考虑升级为带来源/时间戳的结构化对象，但当前不做，避免过度设计。
