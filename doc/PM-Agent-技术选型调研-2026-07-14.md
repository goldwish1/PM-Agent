## 1. 调研目标与范围

### 1.1 项目背景（摘要）

PM Agent 是一个**命令行 Agent**：用户描述项目卡点 → 从约 39 个项目管理工具中推荐 → 对「项目章程 / 风险登记册」多轮起草 → 导出本地 Markdown。产品约束为：

- **练手优先**：从零手写 Agent 循环（感知→推理→行动→观察），对照《从零到一造 Agent》
- **独立新仓库**，自带 `tools.json`（可从 toolbox 拷贝内容，**代码栈不必对齐**）
- **不做**：Web 前端、数据库、登录、长期记忆、多渠道、子 Agent

### 1.2 本次调研要回答的问题

1. 用什么语言 / 运行时做 CLI Agent？  
2. Agent 循环自己写，还是用现成 Agent 框架？  
3. 怎么调大模型、怎么做 Tool Calling？  
4. CLI 交互用什么库？  
5. 工具参数校验、配置与产物落盘用什么？  
6. MVP 要不要数据库？  

### 1.3 在线调研来源（2026-07）

| 主题 | 参考 |
|------|------|
| 何时不该上框架 | Speakeasy：简单 tool loop 可跳过重型框架 |
| Tool 循环官方模式 | Anthropic「Build a tool-using agent」：while + messages 历史 |
| DeepSeek Tool Calls | 官方文档：OpenAI 兼容；Python/Node `openai` SDK + `base_url` |
| TS Agent 栈（对照用） | Vercel AI SDK / OpenAI Agents / Mastra — 本项目练手期不采用重型框架 |

## 2. 技术选型对比分析

### 2.1 语言与运行时（已修订）

| 选项 | 作用 | 优点 | 缺点 | 结论 |
|------|------|------|------|------|
| **Python 3.11+** | CLI、循环、工具、文件 I/O | Agent 教程 / 官方示例多；脚本迭代快；自学成本通常更低；`openai` Python SDK 成熟 | 书籍配套 AgentClaw 是 TS，对照源码要「跨语言读」 | **采用（学习项目推荐）** |
| TypeScript + Node.js | 同上 | 类型严；Cursor 生态熟；若要对齐 Web/toolbox 更顺 | 原先选型理由是「跟 toolbox 对齐」——你已明确**不需要** | 备选（若更想读 AgentClaw 源码再换） |

**通俗解释：为什么学习项目更适合 Python？**

1. **你要学的是 Agent 工程（循环、工具、上下文、容错），不是 Node 工程化。** Python 写「一个 while + tools」噪音更少，更容易把注意力放在书里的概念上。  
2. **官方 Tool Calling 教程、社区「最小 Agent」示例，Python 密度更高**，卡住时更好搜。  
3. **不必和 `pm-toolbox`（Next/TS）对齐**后，TypeScript 最大优势被拿掉；剩下的「类型安全」可以用 Pydantic 补。  
4. **代价**：书配套源码是 TS——可以「概念对齐、实现用 Python」，不必逐行抄仓库。

> 若你个人更熟 TS、或强烈想对照 AgentClaw 一行行读，仍可改回 TS；两者对「会不会做成 Agent」差别不大，差别在学习摩擦力。

### 2.2 Agent 编排层（核心决策，不变）

| 选项 | 作用 | 优点 | 缺点 | 结论 |
|------|------|------|------|------|
| **自研 Agent Loop** | while：调模型 → 解析 tool_calls → 执行工具 → 回写 messages → 直到结束或触顶 | 机制透明；对齐书与 PRD「循环可见」 | 错误处理、上限要自己写 | **采用** |
| LangChain / LangGraph / CrewAI 等 | 框架托管编排 | 出活快 | 练手被框架淹没 | **不采用（MVP）** |
| 各家 Agents SDK 黑盒 Runner | 托管循环 | 省事 | 看不清循环 | 不采用 |

**关键理由**：工具少、流程近似线性 → 自研几十到一两百行 loop 足够，也是本项目的学习价值所在。

---

### 2.3 大模型调用与 Tool Calling

| 选项 | 作用 | 优点 | 缺点 | 结论 |
|------|------|------|------|------|
| **`openai` Python SDK + DeepSeek `base_url`** | Chat Completions + tools | 官方兼容路径清晰；与工具回传语义一致；可切兼容网关 | 模型 ID 可能更新 | **默认采用** |
| `anthropic` Python SDK | Messages + tool_use | 循环教程好 | 多一套消息形状 | 可选备选 Provider |
| 裸 `httpx`/`requests` | 不依赖官方 SDK | 依赖少 | 重试与类型要自己拼 | 不划算 |

**通俗解释**：模型只「提议」调用哪个工具；**你的 Python 代码必须真正执行工具，再把结果作为 `role=tool` 喂回去**。DeepSeek 走 OpenAI 兼容协议，用同一个 `OpenAI(base_url=...)` 客户端即可。

**默认策略**：

- 主：DeepSeek（`DEEPSEEK_API_KEY`）  
- 备：同协议的 OpenAI / 兼容端（换 key 与 base_url）  
- `.env` 存密钥，永不写入导出 Markdown  

---

### 2.4 CLI 交互层

| 选项 | 作用 | 优点 | 缺点 | 结论 |
|------|------|------|------|------|
| **标准库 `input()` 循环** | 读一行、打回复 | 零依赖；最贴最小 Agent | 无花哨 UI | **采用** |
| `prompt_toolkit` / Rich | 更好输入与着色 | 体验更好 | MVP 非必须 | 可选后期 |
| Typer / Click | 子命令 CLI | 适合多命令工具 | 本产品是长对话 | 可仅作启动入口 |

---

### 2.5 工具定义与参数校验

| 选项 | 作用 | 优点 | 缺点 | 结论 |
|------|------|------|------|------|
| **手写 Tool Registry**（name / description / parameters / execute） | 对齐书中工具六字段思路 | 练「说明书」质量 | 要自维 schema | **采用** |
| **Pydantic** | 校验模型传来的 arguments | Python 惯用；防幻觉字段 | 一个依赖 | **采用** |
| MCP | 外挂工具进程 | 可扩展 | MVP 过重 | 不做 |

**建议工具集合（与 PRD 对齐，约 6 个）**：

| 工具名（示意） | 副作用 | 说明 |
|----------------|--------|------|
| `list_tools` / `search_tools` | 无 | 浏览/检索 |
| `recommend_tools` | 无 | 推荐 1～3 |
| `get_tool_detail` | 无 | 详情 |
| `draft_project_charter` | 有 | 章程草稿 |
| `draft_risk_register` | 有 | 风险草稿（1～3 条） |
| `export_markdown` | 有（写盘） | 仅 `output/` |

### 2.6 数据与持久化（不变）

| 选项 | 结论 |
|------|------|
| 内存 messages + 会话草稿 | **采用** |
| SQLite / 向量记忆 | **不做** |
| `tools.json` 只读 + `output/*.md` | **采用** |

| 选项 | 作用 | 结论 |

## 3. 推荐技术栈总表（拟确认 · 已修订）

| 层级 | 选型 | 一句话理由 |
|------|------|------------|
| 语言/运行时 | **Python 3.11+** | 学习摩擦力低；Agent 示例多；不强制对齐 toolbox |
| Agent 编排 | **自研 Loop** + 迭代上限 + `[tool]` 日志 | 练手核心 |
| LLM | `openai` Python SDK → DeepSeek | Tool Calls 成熟、协议统一 |
| 校验 | Pydantic | 拦住错误参数 |
| CLI | 标准库 `input()` | 最小可用 |
| 数据 | `data/tools.json` + 内存会话 + `output/` | 无数据库也能闭环 |
| 配置 | `.env` + `.gitignore` | 密钥安全 |

---

## 4. 关键技术选型理由（汇总）

1. **语言改 Python**：这是学习项目，目标是 Agent 工程能力，不是 Next 栈统一；砍掉「对齐 toolbox」后，Python 更合适。  
2. **仍自研循环**：框架省事但挡视线；书与 PRD 都要求看得见循环。  
3. **仍用 OpenAI 兼容客户端 + DeepSeek**：工具回传语义清晰，Provider 可换。  
4. **Pydantic + output 白名单**：安全挡在工具层，不靠模型自觉。

## 5. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 对照 AgentClaw（TS）吃力 | 以书的概念/流程图为准；实现用 Python 复刻机制即可 |
| 模型乱造工具名 | slug 与 `tools.json` 交叉校验；错误信息写成「下一步指令」 |
| 手写循环写歪 | 对照官方 tool 回传模式；设 `max_iterations` |
| 上下文塞满 tools.json | 先摘要列表，详情再 `get_tool_detail` |
