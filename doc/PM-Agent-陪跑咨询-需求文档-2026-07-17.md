## 1. 问题背景

### 1.1 用户反馈的核心问题

当前 MVP（阶段 0～5）已实现「推荐工具 → 起草文档 → 导出 Markdown」的闭环，但用户在验收对话中提出：**当前流程没有解决用户的真实问题**。

用户的真实使用旅程是决策支持型的多轮对话，而非表单填写：

```
用户：我遇到了一个项目决策难以进行，背景是XXXXXX，我该怎么办？
Agent：我推荐你使用 XXXX 工具，原因是 XXXXXX
用户：那我应该怎么使用这个工具呢？
Agent：使用思路是 XXXXX，结合你的情况，我们要 XXXXX
（用户和 Agent 进行若干轮细节讨论）
用户：好的，帮我起草一下文件
Agent：（根据前面讨论内容整理、提炼）起草文档
```

### 1.2 现状与目标的三个断层

对照现有实现（`recommend_tools`、`get_tool_detail`、`draft_project_charter`、`draft_risk_register`、`prompts.py`），识别出以下断层：

1. **推荐理由是分类学解释，不是咨询建议**：`recommend_tools` 返回的 `reason` 目前是「与卡点匹配，且属于 XX 过程组/XX 知识领域」，是静态归类描述，不是结合用户情境的建议。
2. **"怎么用工具"阶段在架构里没有位置**：用户问"这个工具该怎么用"时，系统当前唯一能接的是 `get_tool_detail`，返回的是工具库里固定的 `steps`/`scenarios`（通用说明书），不会结合用户的具体情境（预算、干系人、时间压力等）给出个性化建议。PRD 现有流程图中「查看工具详情」与「起草」是两条平行分支，中间没有「陪跑讨论」这个中间态。
3. **起草与前面讨论内容脱节**：`draft_project_charter` / `draft_risk_register` 靠 LLM 在对话里逐字段/逐条追问用户，本质是孤立表单收集，不会把用户在陪跑讨论阶段说过的背景、约束、决策依据结构化地提炼进草稿。用户如果在推荐后聊了很多轮方案细节，这些内容不会被保证性地沉淀进最终文档，等同于"讨论白聊了"。

### 1.3 问题的产品定位

MVP 现状更接近「工具推荐器 + 表单填写器」，而用户期望的是「项目决策陪伴 Agent」——起草文档应是深度讨论后的**副产品/记录**，而不是目的本身。

## 2. 需求目标

在不破坏现有 MVP 闭环（推荐 → 起草 → 导出）的前提下，补齐"推荐"与"起草"之间的**陪跑讨论阶段**，并让讨论内容能够结构化沉淀、供起草阶段复用。

### 2.1 核心原则（已与用户逐条确认）

| # | 决策点 | 结论 |
|---|--------|------|
| 1 | "怎么用工具"阶段应给出通用方法论还是个性化建议？ | **个性化建议**：Agent 需主动追问情境细节（预算、干系人、时间压力等），结合工具的 steps/scenarios 给出针对用户情况的使用思路，而不是直接罗列 `steps` 字段。 |
| 2 | 起草时应"自动提炼字段"还是"讨论归讨论，表单归表单"？ | **Agent 主动总结提炼**，但必须保留预览确认闸门：Agent 从讨论中生成候选字段值 → 展示预览 → 用户确认或修正 → 才真正写入草稿/导出。不允许绕过确认直接导出。 |
| 3 | 陪跑讨论阶段要不要是显式的 session 状态，还是完全交给 LLM 自由发挥？ | **需要显式状态（`CONSULTING`）**。纯 prompt 约束在长对话中会随上下文衰减而失效，工程上需要状态机 + 结构化沉淀机制兜底，不能只靠“写更好的 prompt”。 |
| 4 | 陪跑讨论要不要设置最大轮次上限？ | **不设硬上限**。该阶段的价值就在于讨论深度，人为截断会破坏体验。改为在系统提示词中引导 LLM：当讨论已经充分时，主动询问用户是否要起草。 |
| 5 | 用户未经陪跑讨论直接说"帮我起草 XX"，是否强制引导先讨论？ | **不强制**。保留现有"逐字段/逐条问答"兜底路径，允许目的明确、不需要陪聊的用户直接进入起草。 |
| 6 | 这套"讨论 → 沉淀 → 提炼 → 确认"逻辑是否只针对项目章程或风险登记册？ | **不是，是普适机制**。适用于所有 `draftable: true` 的工具，机制统一、不按文档类型重复设计，便于未来扩展第三个可起草工具时复用。 |

### 2.2 功能需求清单

1. **进入陪跑讨论**
   - 用户在收到推荐后，追问"这个工具该怎么用"或表达希望深入讨论时，Agent 进入陪跑讨论阶段（对应新增的 `CONSULTING` 会话状态）。
   - 该阶段仅针对 `draftable: true`（当前为「项目章程」「风险登记册」）的工具触发；其余工具维持现状（仅说明与推荐，不支持起草，也不支持陪跑）。

2. **个性化讨论**
   - Agent 结合被讨论工具的 `steps`/`scenarios` 作为线索，主动追问用户的具体情境（背景、约束、干系人、预算、时间压力等）。
   - 回复内容必须针对用户情境，禁止仅复述工具库中的通用步骤说明。

3. **讨论内容结构化沉淀**
   - 讨论过程中，Agent 需要将判断为「有价值的项目背景信息」结构化记录下来（而不是仅依赖对话历史让模型自己在最后回忆）。
   - 沉淀的内容需能够在后续起草阶段被复用为字段候选值的来源。

4. **陪跑到起草的过渡**
   - 陪跑讨论没有强制轮次上限；由 Agent 判断讨论是否已充分，可主动询问用户"是否可以开始起草"。
   - 用户明确要求起草时，Agent 基于沉淀内容 + 对话历史，自动提炼出对应文档（项目章程/风险登记册）的候选字段值。

5. **提炼结果的确认闸门**
   - Agent 提炼出的候选字段值必须先以预览形式展示给用户。
   - 用户可以对预览中的任意字段进行修正（而不是从空白重新逐项回答）。
   - 用户确认后才允许进入导出。

6. **兜底路径保留**
   - 若用户没有经过陪跑讨论、直接要求起草某个可起草工具的文档，Agent 应继续走现有的逐字段/逐条问答模式，不强制打断或引导先讨论。

7. **普适性**
   - 上述陪跑讨论与提炼确认机制，需以「不区分具体文档类型」的方式设计，保证未来新增可起草工具时无需重新设计流程。

### 2.3 明确不做的事情（边界）

- 不改变现有 `draft_project_charter` / `draft_risk_register` 的字段结构与增量合并语义。
- 不引入陪跑讨论轮次上限、不引入强制陪跑的产品规则。
- 不为项目章程与风险登记册分别设计两套独立的陪跑/沉淀逻辑。
- 不改变导出机制（仍只写本地 `output/`，仍需用户明确"确认导出"）。
- 不在陪跑阶段引入自动导出或跳过预览的能力。

---

## 3. 验收标准

1. 用户在收到工具推荐后，可以就"怎么用"发起追问，Agent 的回复必须包含针对用户已提供情境的具体建议，而非仅罗列工具库中的 `steps`。
2. 在陪跑讨论进行数轮后，用户说"帮我起草"，Agent 生成的预览中应体现讨论过程中提到的关键信息（而非全部字段仍是「待补充」占位）。
3. 用户对预览提出修正意见后，Agent 能正确按用户修正更新对应字段，而不是重新发起整轮逐字段问答。
4. 用户未经陪跑、直接说"帮我起草风险登记册"，Agent 仍可按现有逐条问答方式完成起草（兜底路径不受影响）。
5. 该机制在「项目章程」与「风险登记册」两个场景下行为一致，不存在专属特殊分支逻辑。
6. 现有回归测试（`tests/`）中与推荐、起草、导出相关的用例应继续通过；新增行为需补充对应测试。
